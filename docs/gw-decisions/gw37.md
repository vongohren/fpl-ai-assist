## 🚨 Surprises

### Surprise #1 — Unexpected -4 hit on 3 transfers
- **Squad endpoint reported:** `"free_transfers": 3` going into the deadline
- **FPL applied:** -4 hit on the 3rd transfer (`transfers.cost: 4, limit: 3, made: 3`)
- **Effective FT count:** 2, not 3
- **Net cost:** -4 points on GW37 score
- **Possible causes:**
  - Squad endpoint returned a stale or miscounted FT value
  - FPL's bank-FT cap behaved differently than reported (max 5 banked rules quirk?)
  - Some prior in-week action consumed an FT we didn't account for
- **Action committed before this was visible** — discovered only from the `save_team` response. Unrecoverable without rolling back, which would mean re-doing transfers (also costing).
- **Lesson:** Don't fully trust the squad endpoint's `free_transfers` count. Cross-check by simulating one transfer first, or pull a separate FT-source if available, before committing a 3-FT plan.

### Surprise #2 — Brave Search rate-limited mid-eval
- Captaincy and differentials community-trends queries returned empty (HTTP 429)
- Only the transfers topic completed cleanly
- **Impact:** Captaincy discussion was driven by fixtures + the transfers-topic mentions, not a dedicated captain-poll signal
- **Lesson:** When time-pressured, expect Brave to throttle and have a fallback (sequential vs parallel calls, or stop using community trends mid-late-deadline-day)

### Surprise #3 — Chip plan deviation discovered at deadline
- Memory said `WC34 → BB36 → TC38`, but BB36 was never played
- Likely cause: user did not run `/evaluate-gw` for GW35/36/37 cycles, so the BB call window passed
- Forced a constrained GW37 (mandatory BB) rather than the planned quiet hold
- **Lesson:** Memory captures plans, not actual chip plays. Always re-verify `chips.available` against memory at session start, and if a chip should have already been played but is still available, hard-flag this **before** building transfer plans.
